## 1. Summary

Replace the binary `nonOverlap` confirmation gate with a **continuous ConfirmationScore (0???100)** computed on the confirmation candle. Sweeps with `confScore >= minConfirmScore` create zones; sweeps below threshold are rejected with a sub-score breakdown in the rejection reason string.

### New

Replace the `nonOverlap` binary check with 4 sub-scores ??? weighted sum ??? threshold. The `if confScore >= _minConfirmScore` branch is identical to the old `if nonOverlap` branch. The `else` branch includes the sub-score breakdown in the rejection reason.

```pine
if pendingDir != 0 and time > pSweepTime
    // Helper variables available in this scope:
    //   htfAtr        (computed at function root, line 104)
    //   syminfo.mintick (for safe division)
    confRng = math.max(high - low, syminfo.mintick)
    safeAtr = math.max(htfAtr, syminfo.mintick)

    if pendingDir == DIR_BULL
        // --- Displacement Score (40%) ---
        // How far above the zone top did the confirmation candle close?
        dispDist = close - pZoneTop
        dispScore = math.min(math.max(dispDist / safeAtr * 200, 0), 100)

        // --- Rejection Score (30%) ---
        // How cleanly did the confirmation candle avoid overlapping the zone?
        overlapAmt = math.max(pZoneTop - low, 0)
        zoneH = pZoneTop - pZoneBottom
        rejScore = zoneH > 0 ? math.max(100 - overlapAmt / zoneH * 100, 0) : 100

        // --- FVG Score (20%) ---
        // Is there a Fair Value Gap between sweep and confirm candles?
        fvgGap = low - pZoneTop
        fvgScore = fvgGap > 0 ? math.min(fvgGap / safeAtr * 333, 100) : 0

        // --- Efficiency Score (10%) ---
        // What % of the confirmation candle range is upward movement?
        effScore = (close - low) / confRng * 100

        // --- Weighted sum ---
        confScore = dispScore * 0.40 + rejScore * 0.30 + fvgScore * 0.20 + effScore * 0.10

        if confScore >= _minConfirmScore
            eventDir := DIR_BULL
            eventTop := pZoneTop
            eventBottom := pZoneBottom
            eventSweepTime := pSweepTime
            eventConfirmTime := time
            eventLiquidityScore := pLiquidityScore
            eventConfirmationScore := confScore
        else
            debugFailDir := DIR_BULL
            pRejectReason := "Conf " + str.tostring(math.round(confScore, 0)) + " < " + str.tostring(_minConfirmScore) + ": D:" + str.tostring(math.round(dispScore, 0)) + " R:" + str.tostring(math.round(rejScore, 0)) + " F:" + str.tostring(math.round(fvgScore, 0)) + " E:" + str.tostring(math.round(effScore, 0))

    else if pendingDir == DIR_BEAR
        // --- Displacement Score (40%) ---
        dispDist = pZoneBottom - close
        dispScore = math.min(math.max(dispDist / safeAtr * 200, 0), 100)

        // --- Rejection Score (30%) ---
        overlapAmt = math.max(high - pZoneBottom, 0)
        zoneH = pZoneTop - pZoneBottom
        rejScore = zoneH > 0 ? math.max(100 - overlapAmt / zoneH * 100, 0) : 100

        // --- FVG Score (20%) ---
        fvgGap = pZoneBottom - high
        fvgScore = fvgGap > 0 ? math.min(fvgGap / safeAtr * 333, 100) : 0

        // --- Efficiency Score (10%) ---
        effScore = (high - close) / confRng * 100

        // --- Weighted sum ---
        confScore = dispScore * 0.40 + rejScore * 0.30 + fvgScore * 0.20 + effScore * 0.10

        if confScore >= _minConfirmScore
            eventDir := DIR_BEAR
            eventTop := pZoneTop
            eventBottom := pZoneBottom
            eventSweepTime := pSweepTime
            eventConfirmTime := time
            eventLiquidityScore := pLiquidityScore
            eventConfirmationScore := confScore
        else
            debugFailDir := DIR_BEAR
            pRejectReason := "Conf " + str.tostring(math.round(confScore, 0)) + " < " + str.tostring(_minConfirmScore) + ": D:" + str.tostring(math.round(dispScore, 0)) + " R:" + str.tostring(math.round(rejScore, 0)) + " F:" + str.tostring(math.round(fvgScore, 0)) + " E:" + str.tostring(math.round(effScore, 0))
```

### Score Formula Reference

| Component    | Weight | Bullish Formula                     | Bearish Formula                     | Saturation                            |
| ------------ | ------ | ----------------------------------- | ----------------------------------- | ------------------------------------- |
| Displacement | 40%    | `(close - zoneTop) / ATR * 200`     | `(zoneBottom - close) / ATR * 200`  | 0.5x ATR ??? 100                        |
| Rejection    | 30%    | `max(0, 100 - overlap/zoneH * 100)` | `max(0, 100 - overlap/zoneH * 100)` | No overlap ??? 100, full overlap ??? 0    |
| FVG          | 20%    | `(low - zoneTop) / ATR * 333`       | `(zoneBottom - high) / ATR * 333`   | 0.3x ATR ??? 100                        |
| Efficiency   | 10%    | `(close - low) / range * 100`       | `(high - close) / range * 100`      | Close at high ??? 100, close at low ??? 0 |

## 3. Scope Changes

### Inputs (group "2. Advanced")

```pine
minConfirmScore = input.int(0, "Min Confirmation Score", minval=0, maxval=100, group="2. Advanced",
  tooltip="Zones with ConfirmationScore below this are not created. 0 = create all passing confirmations (V29 behavior). Raise to filter weaker confirmations.")
```

## 5. Impacts

### Positive

- **Continuous quality** ??? zones get a quality score for confirmation, not just binary pass/fail
- **More granular rejection** ??? rejection reason includes "Conf 23 < 50: D:10 R:30 F:0 E:40" instead of just "Wick overlap Bull"
- **Score-based filtering** ??? users can set `minConfirmScore` to 0 (all zones), 50 (medium+), 80 (only best)
- **Same default behavior** ??? `minConfirmScore = 0` rejects nothing new; all previously-passing confirmations still pass

### Risk: Zone count changes with `minConfirmScore > 0`

If default stays 0, this is **zero risk** ??? zone count is identical to before. Raising the threshold explicitly reduces zones.

### Risk: Redundant `rng` computation

`confRng = math.max(high - low, syminfo.mintick)` duplicates the `rng` at line 167's computation on the same bar. However, `rng` is computed AFTER the confirmation block. The two are at different scope points and the variable is cheap to assign. No measurable performance impact.

## 6. Testing

1. **Default behavior (minConfirmScore = 0)**: Same number of zones as V9.2.3 with identical liquidity scores. Verify on replay.
2. **Threshold raising**: Set minConfirmScore = 50. Fewer zones, only the strongest confirmations.
3. **Rejection reason**: Enable debug markers. Rejected sweeps show "Conf X < Y: D:... R:... F:... E:..." instead of "Wick overlap".
4. **Dashboard**: ConfirmationScore displayed in zone labels alongside LiquidityScore.
5. **Non-repainting**: All scoring uses current HTF candle's OHLC + `pZoneTop/pZoneBottom` from sweep ??? no lookahead.
